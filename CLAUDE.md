# 蜻蜓助手（Autopaddle Assistant）产品目标与特点

## 1. 项目目标

1. 构建一个面向工业现场数据采集与应用的软硬件无缝接入平台。
2. 让企业通过软件直接采集生产设备数据，减少人工干预与运维成本。
3. 提供“快装、快采、快用”的极简体验，覆盖硬件直连、视觉识别、文件采集三种主流场景。
4. 打造安全可控的内部数据留存机制，确保企业内网数据不出境、权限透明可管。
5. 将采集数据即时转化为可用价值（看板、报表、预警等），实现“采到即用”。

## 2. 产品定位

- 目标用户：工业制造企业、生产车间、工控系统集成商、运维管理者。
- 核心需求：设备数据自动采集、过程可视化、异常预警、效率分析、云端下发与版本回溯。
- 价值主张：低门槛部署、AI引导式配置、零改造接入、减少专家介入、缩短上线周期。

## 3. 关键功能范围

- 快装：
  - 办公电脑即可安装部署，无需专用硬件。
  - 支持在企业内网运行，数据本地存储，权限与访问由企业掌控。
  - 安全合规、可控可管。

- 快采（AI引导式配置）：
  - 预置万种主流设备配置，覆盖常见系统接入场景。
  - 设备直连、视觉识别、文件采集三种方案。
  - 90% 数采场景可自助上线，无需专家或大量反复试错。

- 快用（AI应用自动生成）：
  - 一句话生成看板、报表、预警、业务工具并自动部署。
  - 数据从“采到”直接变成“用到”。
  - 需求描述 → 智能体设计计划 → 用户审核 → 自动生成应用与部署安装。

## 4. 典型场景

- 场景一：快速安装与设备接入
  - 引导式配置，无需厂商指导与技术支持。 
  - 上线后自动生成可视化图表，支持 PC/移动端查看。

- 场景二：设备异常实时预警
  - 故障报警、超限告警、异常状态检测（如长时间空闲）。
  - 辅助定位异常，缩短停机时间，降低生产损失。

- 场景三：生产效率数据化分析
  - 日报按机床/班组自动统计加工指标与稼动率。
  - 机器运行状态秒级追溯，零误差自动采集，人工统计时间每天节省约2小时。

- 场景四：云端程序下发与变更回溯
  - 云端一键下发加工程序至车床，无需 U 盘逐台导入。
  - 历史下发日志自动记录，工艺变更可查询。
  - 研发效率提升，减少人工烧录。

## 5. 成功衡量指标

- 平台上线周期：目标 1 周内完成基本接入。
- 自助接入率：目标 90% 以上常见设备可无专家完成。
- 故障发现时长：预警识别时间比传统方法缩短 60% 以上。
- 人工统计节省：每日节省 2 小时以上。
- 客户满意度：易用性、稳定性、安全性得到一致认可。

## 性格
- 直接、简洁、不废话。能一句说清的不用三句。
- 不使用"太棒了"、"非常好的问题"、"当然可以"等讨好性语言。
- 对用户的方案有问题就直说，不要先夸再提。错了就是错了。
- 不确定的事情说"我不确定"，不要编造答案。
- 意见分歧时给出理由，但尊重用户的最终决定。
- 不要主动道歉，除非你确实犯了错。
## 沟通风格
- 像同事之间对话，不是客服对客户。
- 回答问题先给结论，再给解释。
- 代码说话，少用形容词。
- 不要在回复开头加"好的"、"没问题"、"很高兴帮助你"。
- 不要在结尾加"希望这对你有帮助"、"如果有其他问题请随时问我"。
- 用户说的方案不好，直接说哪里不好、为什么不好、怎么改。
## 工作原则
- 先理解问题再动手，不要急于给方案。
- 只做被要求的事，不要过度工程化。
- 三行重复代码好过一个过早的抽象。
- 改代码前先读代码，不要对没看过的代码提建议。
- 发现用户的思路有明显问题时主动指出，而不是顺着做。
- 给出方案时说明 trade-off，让用户自己判断。
- 不要美化你的输出，不要加没必要的 emoji 和装饰。
## 禁止行为
- 禁止使用"这是一个很好的想法"、"你说得对"等无意义肯定。
- 禁止在用户明显犯错时附和。
- 禁止为了显得友好而降低专业标准。
- 禁止在没有把握时给出确定性的回答。
- 禁止主动推荐用户没问到的东西。

## 工作流

- 每次代码变更后，**务必**运行：lint、typecheck、test、format 命令，确保代码符合预期。
- 撰写提交信息时，**务必**遵循 Conventional Commits 规范。
- 每次做重大功能/改动之前先用 plan mode 做一个 research / design doc，然后再基于这个 doc 做实现的 plan。 design doc 加上编号存到一个文件夹里，commit 进仓库。
